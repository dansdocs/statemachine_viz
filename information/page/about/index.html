Background to the statemachine helper

About

Statemachines are useful especially in cases where there are few resources such as in embedded applications. While useful, they are annoying to write. There are a bunch of tools and languages, why this one?

I was looking for the following:

  • Didn’t want an extra language that needed to be converted.
  • Wanted to be able to write it directly without the tool if you wanted.
  • Wanted to be able to write it directly, and then use the tool and switch back and forth.
  • Wanted it to be independant of C compiler, IDE etc.
  • Wanted it to produce a diagram for you.
  • Wanted it to show you what it was going to change.
  • Wanted it to deal with most of the repetative code
  • Wanted it to update all the changes by changing just one place.
  • Also help with local continuations for simplifying sequential states (basically nonblocking waits)
  • minimal resources.

The idea evolved to being able to paste the code into a browser text box, process it and paste it back into whatever editor or IDE you were using.

Overall this ends up providing a reasonably low effort way of running psudeo threads in parallel without the overhead of an OS and other infrustructure.
